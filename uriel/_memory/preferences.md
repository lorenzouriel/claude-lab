# Preferences

> Voice, tone, and style. Skills read this before generating any content.

## Voice analysis

- **Sentence length:** Short to medium — punchy opener, explanatory follow-through
- **Formality:** Conversational practitioner — direct without being casual, zero corporate tone
- **Person:** First-person throughout
- **Energy:** Confident builder identity — ships things, talks to peers not students
- **Language:** Bilingual — PT-BR for LinkedIn/Instagram, English for Medium/Substack; never mix in the same piece
- **Structure:** Arrow lists (→) in short-form posts; numbered steps + section headers in long-form; "But..." pivots to create tension before the solution
- **Narrative spine:** Every technical post has a before/after story — the problem exists before any code is shown
- **Code-heavy:** Technical posts include real, runnable code blocks with realistic examples
- **Open source identity:** Ships things publicly; references GitHub repos naturally, not as a marketing move

## Writing example

> "Eu montei o Data Eyes pra acabar com aquela bagunça de scripts espalhados. Quem é DBA sabe do trabalho."

> "Communication is always the companies nightmare... I couldn't do anything… until I decided to do something."

## Always avoid

- Anything that looks AI-generated — run through humanizer skill before any output
- Emoji spam (use sparingly, only when the post platform and tone warrant it)
- Corporate buzzwords: "alavancar", "synergy", "delve", "holistic", "leverage", "seamless", "unlock", "game-changer"
- Hedging language: "it might be worth considering", "perhaps", "one could argue"
- Filler openings: "In today's fast-paced world...", "Have you ever wondered..."
- Passive constructions that hide the actor
- Rule of three for its own sake
- Inflated symbolism or over-metaphor

## Style notes

- Always humanize content before final delivery — use `/humanizer` skill
- Posts must feel written by a practitioner talking to peers, not a content creator performing expertise
- Long-form: PMBOK-style structured headers, real folder/code structure in examples, honest about struggles
- Short-form: starts with the pain (DBA knows the struggle), drops straight into the solution, one arrow-listed breakdown, ends with a quiet CTA or question
